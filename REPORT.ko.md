# Skill 실행 벤치마크 — 보고서

**작성일:** 2026-06-20
**대상 모델:** Claude Opus, Claude Haiku
**작업(task):** 작업 레코드 정규화 (자세한 정의는 [`AGENTS.md`](./AGENTS.md))
**설계:** 4개 실행 모드 × 3 trial × 5 케이스 = 모델당 **60개 채점 출력**, 총 **120개**

## 1. 질문

> LLM 에이전트가 Skill을 사용할 때, **문서만(doc-only) / 인라인 코드(inline-code) /
> Python 스크립트(python-script) / Go 바이너리(go-binary)** 중 어떤 패키징 방식이
> 가장 신뢰성 있고 효율적인가?

4개 모드는 *동일한* 정규화 로직을 서로 다른 방식으로 포장합니다. 작업은 고정하고
패키징만 바꿔, 실제 서브에이전트가 얼마나 정확하고(정확성) 얼마나 빠르고 저렴하게
(속도·토큰) 수행하는지 측정했습니다.

## 2. 방법

각 (모드, trial)마다 서브에이전트를 공정성 통제 프롬프트로 디스패치했습니다.

- 해당 모드의 `skills/<mode>/SKILL.md` **만** 읽기 허용.
- 차단: `datasets/tasks.jsonl`(정답 포함), `harness/normalize.py`(참조 구현),
  다른 모드의 SKILL, 기존 트레이스.
- 5개 케이스의 **입력만** 프롬프트에 인라인 제공(정답 `expected`는 절대 미노출).
- 모드별 수행 방식 강제: doc-only는 코드 실행 금지(손으로 추론), inline-code는
  SKILL 내장 코드 사용, python-script는 `transform.py` 실행, go-binary는 빌드 후 실행.
- 출력은 순수 JSON 객체 `{case_id: [...]}` 만 반환.

채점은 `harness/agent_eval.py`가 데이터셋 `expected`와 결정적으로 비교합니다
(객체 키 순서 무관, 배열 순서 유관). 속도·토큰·도구 호출 수는 각 run의
`duration_ms`, `tokens`, `tool_uses` 필드에서 집계합니다.

> 참고: 기계적 `make bench`는 실제 코드 경로를 실행하므로 항상 통과합니다. 이
> 보고서는 그 대신 **실제 에이전트 행동**을 측정하며, 그것이 이 프로젝트의 본래
> 목적입니다.

## 4. 해석

**정확성이 동률(모두 100%)일 때, 효율은 패키징에 따라 뚜렷이 갈립니다.**

1. **doc-only가 가장 빠르고 저렴하다.** 외부 실행 없이 추론만 하므로 도구 호출이
   1회뿐이고, 속도(11s/18s)와 토큰 모두 최소입니다. 이 단순 작업에서는 코드를
   돌리는 비용이 오히려 순수 추론보다 비쌌습니다.
2. **python-script가 가장 무겁다.** 케이스마다 stdin 파이프로 스크립트를 호출하며
   도구 왕복이 8~10회로 늘어, 속도(35s/48s)가 가장 느립니다.
3. **go-binary는 빌드 1회 후 빠르다.** 바이너리를 한 번 빌드한 뒤 실행하므로
   python-script보다 효율적입니다(25s/30s).
4. **모델 간 차이.** Haiku는 Opus보다 토큰을 약 30% 적게 쓰지만(추론 길이가 짧음),
   같은 작업을 더 느리게 수행했습니다(처리량 차이).

## 5. 결론

- **이 난이도에서 패키징은 정확성을 가르지 못한다.** Opus·Haiku 모두 4개 모드에서
  120/120 통과. 변별의 핵심은 **정확성이 아니라 효율(속도·토큰·도구 호출)** 이었다.
- **정확성이 보장되는 한, doc-only가 가장 경제적이다.** 단순·결정적 변환에는 코드를
  포장하기보다 명확한 문서 절차가 더 빠르고 저렴하다.
- **그러나 이는 "보험" 관점에서 뒤집힌다.** 작업이 어려워져 doc-only의 손계산이
  틀리기 시작하면, python-script·go-binary는 *올바른 도구를 호출하기만 하면* 정확성을
  유지한다. 즉 **난도가 높아질수록 외부 실행 패키징의 정확성 안정성이 비용을 정당화**한다.

## 6. 한계 및 다음 단계

Haiku에서도 천장에 막혔다는 것은 **작업 난이도가 변별의 병목**임을 뜻합니다.
패키징 간 정확성 차이를 드러내려면:

- **더 어려운 데이터셋**: 중첩 구조, 대량 레코드, 모호하거나 충돌하는 status 라벨,
  유니코드·적대적 공백 등. doc-only가 먼저 무너지는 지점을 만들어야 한다.
- **trial 확대**: 변별이 나타난 뒤 신뢰 구간을 좁히기 위함.

채점·메트릭 인프라(`harness/agent_eval.py`)와 공정성 프로토콜은 위 후속 실험에
그대로 재사용할 수 있습니다.

## 7. 재현 방법

```bash
make build-go                                   # go-binary 모드용 바이너리 빌드
# 모드/trial별 서브에이전트 디스패치 — 오케스트레이터가 수행(방법 절 참고)
# 각 run은 outputs/agent_runs[_haiku]/<mode>-trial<N>.json 으로 저장

# Opus 채점
python -m harness.agent_eval --runs-dir outputs/agent_runs
# Haiku 채점
python -m harness.agent_eval --runs-dir outputs/agent_runs_haiku \
  --trace outputs/traces/agent-bench-haiku.jsonl
```

원본 에이전트 출력(`outputs/agent_runs*/`)은 비결정적이라 git에서 제외됩니다.
이 보고서의 수치가 본 실행의 공식 기록입니다.

# 후속: 하드 케이스 변별력 실험

위 실험(5 쉬운 케이스)이 천장에 막혀 정확성으로 패키징을 가르지 못했으므로, **손으로
적용하기 극도로 번거로운 하드 케이스셋**(`datasets/tasks-hard.jsonl`, 6케이스)을 만들어
다시 측정했다. 계약·구현은 그대로 두고 **입력만** 어렵게 했다(대량 레코드, 까다로운
수치 정렬 `1·2·10·21·100`, 매핑외 status 라벨 `blocked/review/on hold`, 선택 필드 다수,
내부 이중공백 `"Spaced  Out"`, 중복 id, 탭 공백). expected는 참조 구현으로 생성해 코드
3모드의 통과를 보장했다. 설계: 4모드 × 3 trial × 6케이스 × **2모델(Opus·Haiku)**.

## 하드 정확성 (모델 × 모드) — 변별 발생

| 모드 | Opus | Haiku |
|------|-----:|------:|
| doc-only | **100%** (18/18) | **77.8%** (14/18) |
| inline-code | 100% (18/18) | 100% (18/18) |
| python-script | 100% (18/18) | 100% (18/18) |
| go-binary | 100% (18/18) | 100% (18/18) |

**Haiku doc-only만 무너졌다.** 실패 지점:
- **hard-006 → 0/3** (3회 모두 실패): `"  Spaced  Out  "`를 `"Spaced Out"`으로 — 양끝뿐
  아니라 **내부 이중공백까지** 축소. 손 trim의 전형적 실수.
- **hard-003 → 2/3** (1회 실패): `Completed→done`을 한 번 `doing`으로 오매핑.

코드 실행 3모드는 같은 입력·같은 약한 모델에서도 **전부 100%** — 코드가 trim·매핑을
정확히 수행하기 때문. 즉 **동일 모델(Haiku)·동일 입력인데 doc-only는 틀리고
inline/python/go는 맞는** 직접 대조가 잡혔다.

## 하드 속도·토큰 (trial 평균)

| 모드 | Opus 속도 | Haiku 속도 | Opus 토큰 | Haiku 토큰 | Opus 도구 | Haiku 도구 |
|------|---------:|----------:|---------:|----------:|--------:|---------:|
| doc-only | **22s** | **25s** | 43,627 | 32,915 | 1 | 1 |
| inline-code | 44s | 56s | 47,428 | 37,081 | 3 | 3 |
| python-script | 69s | 69s | 49,562 | 35,571 | 14 | 9 |
| go-binary | 57s | 72s | 47,146 | 35,918 | 6 | 9 |

효율 순위는 쉬운 셋과 동일: doc-only가 가장 빠르고 도구 호출 최소, python-script가 가장 무거움.

## 종합 결론

**패키징 변별력은 (모델 강도) × (task 난이도)의 상호작용으로 결정된다.**

| | 쉬운 task | 어려운 task |
|--|----------|-----------|
| **강한 모델(Opus)** | 전모드 100% (변별 없음) | 전모드 100% (변별 없음) |
| **약한 모델(Haiku)** | 전모드 100% (변별 없음) | **doc-only 77.8% vs 코드 100% (변별!)** |

- **코드 실행 모드(inline/python/go)는 4개 조건 모두에서 100%** — 모델·난이도와 무관한
  정확성 안정성을 제공한다.
- **doc-only는 (약한 모델 × 어려운 task)에서만 무너진다.** 강한 모델이거나 쉬운 task면
  doc-only도 충분하고, 오히려 가장 빠르고 저렴하다.
- 따라서 실무 지침: **모델이 약하거나 task가 까다로울수록, 실행 로직을 스크립트·바이너리로
  포장(외부 실행)하는 것이 정확성 보험으로 비용을 정당화한다.** 반대로 강한 모델·단순 task에서는
  doc-only가 효율 면에서 우위다.

# 후속 2: 어려운 동작 실험 — 의존성 위상정렬

앞 두 실험은 모두 같은 동작("정규화")에서 입력만 어렵게 했고, 강한 모델(Opus)은 입력이
아무리 까다로워도 doc-only로 100%를 유지했다. 이번엔 **연산 자체가 알고리즘적으로 깊은**
새 동작 — **의존성 위상정렬**(Kahn 알고리즘, 가용 노드 중 id 오름차순, 사이클 탐지)을
추가했다(`datasets/tasks-toposort.jsonl`, 6케이스, 5~12노드). 데이터셋은 위상순이 단순
id정렬과 **다르도록** 설계해(예: `["5","4","3","2","1"]`), "그냥 id순 출력" 휴리스틱으로는
못 맞추게 했다. 계약은 참조 구현으로 expected를 생성해 코드 3모드의 통과를 보장했다.
설계: 4모드 × 3 trial × 6케이스 × 2모델(Opus·Haiku).

## 위상정렬 정확성 (모델 × 모드) — 강한 모델도 무너짐

| 모드 | Opus | Haiku |
|------|-----:|------:|
| doc-only | **83.3%** (15/18) | **88.9%** (16/18) |
| inline-code | 100% (18/18) | 100% (18/18) |
| python-script | 100% (18/18) | 100% (18/18) |
| go-binary | 100% (18/18) | 100% (18/18) |

**이번엔 Opus doc-only도 무너졌다 — 그것도 Haiku보다 더 낮게.** 실패는 전부 큰 그래프
(topo-005: 12노드, topo-006: 9노드)에 집중됐다. Opus doc-only는 세 trial 모두에서 큰
그래프를 하나씩 틀렸다(노드 indegree를 손으로 추적하다 ready 노드를 놓치거나 출력에서
한 노드를 누락). 작은 그래프(topo-001~004)는 두 모델 모두 항상 맞혔다.

코드 실행 3모드는 두 모델 모두 **전부 100%** — 알고리즘을 코드로 돌리므로 그래프 크기와
무관하게 정확하다.

## 위상정렬 속도·토큰 (trial 평균)

| 모드 | Opus 속도 | Haiku 속도 | Opus 토큰 | Haiku 토큰 |
|------|---------:|----------:|---------:|----------:|
| doc-only | 22s | 26s | 46,586 | 32,536 |
| inline-code | 25s | 35s | 48,835 | 34,979 |
| python-script | 33s | 41s | 48,490 | 35,309 |
| go-binary | 26s | 40s | 47,743 | 33,900 |

## 전체 종합 — doc-only 정확성 3실험 비교

| 실험 (동작/난이도) | Opus doc-only | Haiku doc-only | 코드 3모드 (양 모델) |
|--------------------|:------------:|:-------------:|:------------------:|
| 정규화 · 쉬움 | 100% | 100% | 100% |
| 정규화 · 하드 입력 | 100% | 77.8% | 100% |
| **위상정렬 (깊은 동작)** | **83.3%** | **88.9%** | **100%** |

**결론: 변별의 가장 강력한 축은 입력 난이도가 아니라 '동작의 알고리즘적 깊이'다.**

- 입력만 어렵게 하면 약한 모델만 무너진다(Haiku 정규화 하드 77.8%, Opus는 100% 유지).
- 그러나 **연산 자체가 깊으면(위상정렬) 강한 모델의 doc-only조차 무너진다**(Opus 83.3%).
  모델 강도는 doc-only를 구원하지 못했다 — Opus가 Haiku보다 오히려 낮았다.
- **코드 실행 모드(inline/python/go)는 6개 (모델×실험) 조합 전부에서 100%**였다. 이것이
  핵심 가치다: 실행 로직을 코드로 포장하면 모델·난이도·동작 깊이와 무관하게 정확성이 보장된다.
- 실무 지침(갱신): **동작이 단순 매핑/정렬을 넘어 다단계 알고리즘(그래프·상태기계·누적
  계산 등)에 이르면, doc-only는 강한 모델로도 신뢰할 수 없다. 그 영역에서는 스크립트·
  바이너리 포장이 선택이 아니라 필수다.**
