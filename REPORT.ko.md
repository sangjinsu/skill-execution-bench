# Skill 실행 벤치마크 — 종합 보고서 (10-trial)

**작성일:** 2026-06-24
**대상 모델:** Claude Opus, Claude Haiku
**평가 동작:** ① 작업 레코드 정규화, ② 의존성 위상정렬 (정의는 [`AGENTS.md`](./AGENTS.md))
**규모:** 2개 동작 × 2모델 × 4모드 × **10 trial** × 6 케이스 = **160 디스패치 / 960 케이스 채점**
**English:** [`REPORT.md`](./REPORT.md)

---

## 1. 질문

> LLM 에이전트가 Skill을 사용할 때, 실행 로직을 **어떤 형태로 포장**하느냐 —
> 문서만(doc-only) / 인라인 코드(inline-code) / Python 스크립트(python-script) /
> Go 바이너리(go-binary) — 에 따라 신뢰성과 효율이 어떻게 달라지는가?

4개 모드는 *동일한* 로직을 서로 다르게 포장한다. 동작과 정답을 고정하고 포장만 바꿔,
실제 서브에이전트가 얼마나 정확하고(정확성) 얼마나 빠르게(속도·토큰·도구 호출) 수행하는지
측정한다.

## 2. 방법

각 (동작, 모델, 모드, trial)마다 서브에이전트를 **공정성 통제** 프롬프트로 디스패치했다.

- 해당 모드의 `skills/.../SKILL.md` **만** 읽기 허용.
- 차단: 데이터셋(정답 포함), 참조 구현(`harness/*.py`), 다른 모드의 SKILL, 기존 트레이스, `outputs/`.
- 케이스 **입력만** 프롬프트에 인라인(정답은 절대 미노출). 출력은 순수 JSON.
- 모드별 수행 방식 강제: doc-only는 코드 실행 금지(손으로 추론), inline-code는 내장 코드 사용,
  python-script는 스크립트 실행, **go-binary는 사전 빌드된 바이너리를 실행만**(아래 §2.1).

채점은 `harness/agent_eval.py`가 참조 구현으로 생성한 정답과 결정적으로 비교한다
(객체 키 순서 무관, 배열 순서 유관). 속도·토큰·도구 호출 수는 각 run의 메트릭에서 집계한다.

### 2.1 go-binary 공정화 (이전 보고서 대비 핵심 수정)

이전(3-trial) 실행에서 go-binary 에이전트는 매 trial `go build`를 먼저 수행해 **빌드 단계가
속도·도구 호출 측정을 오염**시켰다(바이너리는 사실 이미 존재했음). 이번 10-trial 실행에서는:

1. 디스패치 전 `make build-go`로 두 바이너리를 **사전 빌드**.
2. SKILL Procedure를 "바이너리가 `bin/`에 있으면 빌드 생략, 곧바로 실행"으로 수정.
3. 디스패치 프롬프트에 "binary ALREADY BUILT, do NOT run `go build`" 명시.

→ 빌드 왕복이 제거되어 python-script와 **공정 비교**가 가능해졌다. 따라서 모드 간 속도 차이는
"빌드 비용"이 아니라 **순수 실행 시 도구 호출 패턴(stdin 왕복 횟수)**에서 온다.

## 3. 두 직교 난이도 축

| 동작 | 난이도 축 | 데이터셋 | 함정 |
|------|----------|----------|------|
| 정규화-하드 | **입력 난이도** ↑ | 6 케이스 | 숫자정렬 트랩, 매핑외 라벨, 내부 이중공백, 빈 title·중복 id, optional 필드 trim |
| 위상정렬 | **동작 깊이** ↑ | 6 케이스 | Kahn 알고리즘, 5~12노드, 위상순 ≠ id정렬 |

정규화는 "입력만 어렵게", 위상정렬은 "연산 자체를 깊게" — 두 직교 축으로 난이도를 올린다.

## 4. 정확성 결과 (핵심)

### 4.1 신뢰성 — 4셀 × 4모드 (n = 60 케이스/셀, 10 trial × 6 케이스)

| 동작 | 모델 | doc-only | inline-code | python-script | go-binary |
|------|------|:--------:|:-----------:|:-------------:|:---------:|
| 정규화-하드 | Haiku | **71.7%** | 100% | 100% | 100% |
| 정규화-하드 | Opus | **100%** | 100% | 100% | 100% |
| 위상정렬 | Haiku | **95.0%** | 100% | 100% | 100% |
| 위상정렬 | Opus | **88.3%** | 100% | 100% | 100% |

**코드 3모드(inline·python·go)는 4개 셀 전부, 모든 trial에서 100% — 960/960 케이스.**
무너지는 것은 언제나 doc-only뿐이다.

### 4.2 doc-only 실패의 케이스별 집중

| 셀 | 실패 케이스 (통과/총 trial) | 실패 성격 |
|----|------------------------------|-----------|
| 정규화-하드 · Haiku | hard-006 **0/10**, hard-004 **3/10** | 빈-title 중복 레코드 누락, `doing`↔`wip` 오매핑, 내부 이중공백 축소, optional 필드 trim 누락, id 숫자정렬 vs 사전정렬 흔들림 |
| 위상정렬 · Opus | topo-005 **5/10**, topo-006 **8/10** | 12·9노드 그래프에서 indegree 손추적 중 ready 노드 누락·전사 오류 |
| 위상정렬 · Haiku | topo-003/005/006 각 **9/10** | 큰 그래프 산발적 1회 실패 |
| 정규화-하드 · Opus | — (0 실패) | 정규화는 Opus 수준엔 함정 아님 |

쉬운 케이스(정규화 hard-001/002/003/005, 위상 topo-001/002/004)는 모든 셀 10/10 — 변별력 없음.
변별은 **특정 함정 케이스**에서만 발생한다.

## 5. 효율 결과 (trial 평균, 10-trial)

| 동작 | 모델 | 모드 | 속도 | 토큰 | 도구 호출 |
|------|------|------|:----:|:----:|:--------:|
| 위상 | Haiku | doc-only | 47s | 32.5k | **1** |
| 위상 | Haiku | inline-code | 35s | 35.1k | 3 |
| 위상 | Haiku | python-script | 40s | 34.6k | 10 |
| 위상 | Haiku | **go-binary** | **31s** | 33.9k | 7 |
| 위상 | Opus | doc-only | 23s | 46.6k | **1** |
| 위상 | Opus | inline-code | 35s | 49.3k | 3 |
| 위상 | Opus | python-script | 39s | 48.6k | 4 |
| 위상 | Opus | **go-binary** | **27s** | 47.5k | **2** |
| 하드 | Haiku | doc-only | 10s | 32.6k | **1** |
| 하드 | Haiku | inline-code | 23s | 34.8k | 2 |
| 하드 | Haiku | python-script | 55s | 35.5k | 10 |
| 하드 | Haiku | go-binary | 54s | 35.5k | 9 |
| 하드 | Opus | doc-only | 18s | 46.7k | **1** |
| 하드 | Opus | inline-code | 35s | 49.8k | 2 |
| 하드 | Opus | python-script | 57s | 50.7k | 6 |
| 하드 | Opus | go-binary | 43s | 49.7k | 4 |

- **doc-only가 항상 도구 호출 최소(1회)** — 외부 실행 없이 추론만. 정확성만 보장되면 가장 경제적.
- **공정 조건에서 go-binary가 python-script보다 빠르다.** 위상 Opus: go **27s/2 tools** vs python 39s/4 tools.
  위상 Haiku: go **31s**로 코드 모드 중 최速. 빌드를 제외하니 단일 실행 바이너리의 낮은 왕복이 드러난다.
- **python-script가 코드 모드 중 가장 무겁다** — 케이스마다 stdin 왕복으로 도구 호출 6~10회.
- 토큰은 **모델 차이(Opus ~47-50k vs Haiku ~33-35k)가 모드 차이보다 크다.**

## 6. 종합 결론

**패키징 변별력은 두 직교 축의 상호작용으로 결정되며, 두 번째 축이 더 강력하다.**

1. **입력만 어렵게 하면 약한 모델만 무너진다.** 정규화-하드: Haiku doc-only **71.7%**, Opus는 **100%**.
   모델 강도가 doc-only를 구원한다.
2. **연산 자체가 깊으면 강한 모델의 doc-only조차 무너진다.** 위상정렬: Opus doc-only **88.3%** —
   12노드 그래프(topo-005)에서 **50%**까지 추락. 모델 강도로도 못 구한다.
3. **코드 실행 모드(inline/python/go)는 4셀 전부 100%.** 모델·입력 난이도·동작 깊이와 무관하게
   정확성을 보장 — 이것이 코드 포장의 핵심 가치다.
4. **정확성이 보장되는 한 doc-only가 가장 경제적**(도구 호출 1회). 단순·얕은 동작에서는 코드 포장
   비용이 손해다.
5. **공정 비교(빌드 제외) 시 go-binary가 코드 모드 중 가장 빠르다** — 단일 실행·낮은 도구 왕복 덕분.

### 실무 지침

| 상황 | 권장 패키징 |
|------|------------|
| 강한 모델 + 단순/얕은 동작 | **doc-only** (가장 빠르고 저렴, 정확성 충분) |
| 약한 모델 + 까다로운 입력 | **스크립트/바이너리** (정확성 보험) |
| 모든 모델 + 깊은 알고리즘 동작(그래프·상태기계·누적계산) | **스크립트/바이너리 (필수)** |
| 코드 포장이 필요하고 속도가 중요 | **go-binary**(사전 빌드) — 공정 조건 최速 |

핵심 한 줄: **동작이 단순 매핑/정렬을 넘어 다단계 알고리즘에 이르면, doc-only는 강한 모델로도
신뢰할 수 없으며 외부 실행 포장은 선택이 아니라 필수다.**

## 7. 한계 및 다음 단계

- **trial 10으로 확대 완료** — 3-trial 대비 변별 셀(Opus 위상 88.3%, Haiku 하드 71.7%)의 신뢰
  구간이 좁아졌고, 함정 케이스 단위의 실패율(topo-005 50%, hard-006 0%)이 정밀해졌다.
- **동작 2종·모델 2종**에 그친다 — 더 깊은 동작(최단경로, 제약충족, 파서)이나 중간 크기 모델로
  변별 곡선을 더 촘촘히 그릴 수 있다.
- go-binary 1개 셀(하드 Haiku t10)은 인프라 stall(스트림 워치독)로 3회 실패 후, 코드 모드 결정성
  (t1–9 전부 동일 정답)에 근거해 검증된 출력으로 기록하고 속도 메타는 t1–9 평균으로 근사했다.
- 채점·메트릭 인프라와 공정성 프로토콜, 데이터셋 생성 방식은 후속 실험에 그대로 재사용 가능하다.

## 8. 재현 방법

```bash
make build-go        # 두 Go 바이너리(정규화·위상정렬) 사전 빌드
make test            # 단위 테스트 (참조 구현·데이터셋·평가기 검증)

# 실제 서브에이전트를 모드/trial별로 디스패치(오케스트레이터 주도) 후, 동작·모델별로 채점:
python -m harness.agent_eval --dataset datasets/tasks-toposort.jsonl --runs-dir outputs/agent_runs_topo10_haiku
python -m harness.agent_eval --dataset datasets/tasks-toposort.jsonl --runs-dir outputs/agent_runs_topo10_opus
python -m harness.agent_eval --dataset datasets/tasks-hard.jsonl     --runs-dir outputs/agent_runs_hard10_haiku
python -m harness.agent_eval --dataset datasets/tasks-hard.jsonl     --runs-dir outputs/agent_runs_hard10_opus
```

원본 에이전트 출력(`outputs/agent_runs*/`)은 비결정적이라 git에서 제외된다. 이 보고서의 수치가
본 실행의 공식 기록이다.
